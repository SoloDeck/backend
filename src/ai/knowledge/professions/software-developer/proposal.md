# Software Development Proposal Guide

## 1. Purpose

Defines profession-specific guidance for generating software development proposals from qualified leads. This guide supplements the proposal generation framework and does not define proposal formatting.

---

## 2. Supported Project Types

Generate proposals for engagements including:

- Business websites
- Web applications
- Mobile applications
- E-commerce systems
- Customer portals
- Internal business systems
- CRM / ERP modules
- Booking systems
- REST APIs
- Backend services
- Workflow automation
- SaaS platforms
- Third-party integrations

---

## 3. Proposal Structure

A software development proposal should generally include:

1. Project summary
2. Scope of work
3. Deliverables
4. Timeline
5. Pricing
6. Assumptions
7. Exclusions

---

## 4. Scope Interpretation

### Converting Client Requirements into Scope

Convert qualified intake information into implementation scope.

| Intake Information | Proposal Scope |
|-------------------|----------------|
| Business goal | Project objective |
| Project type | Primary deliverable |
| Target users | Intended system users |
| Platforms | Deployment platforms |
| Core features | Functional scope |

Group related features into logical software modules where appropriate.

---

### Handling Missing Information

If important information is unavailable:

- State it as an assumption or clarification item.
- Do not invent functional requirements.
- Limit the proposal to confirmed project scope.

---

### Handling Assumptions

Distinguish between:

- Confirmed requirements
- Reasonable professional assumptions
- Recommendations requiring client approval

Assumptions should never be presented as confirmed requirements.

---

## 5. Deliverable Guidance

### Typical Deliverables

Depending on project scope, deliverables may include:

- Website or web application
- Mobile application
- REST API
- Backend services
- Database implementation
- Administrative dashboard
- Technical documentation
- Deployment configuration

---

### Deliverable Wording

Describe deliverables as measurable outputs rather than implementation tasks.

Prefer:

- Responsive business website
- Customer management module
- Online booking system

Avoid:

- Build frontend
- Write backend code
- Implement database

---

### Optional Deliverables

Only include optional deliverables when supported by the inquiry or profession-specific pricing.

Examples:

- Deployment assistance
- User documentation
- Administrator training
- Maintenance period
- Post-launch support

---

## 6. Timeline Guidance

### Common Project Phases

Typical phases include:

- Requirements confirmation
- Design (if applicable)
- Development
- Testing
- Deployment

---

### Timeline Estimation Guidance

Estimate timelines according to:

- Project scope
- Feature complexity
- Number of platforms
- Third-party integrations

Do not fabricate precise schedules when insufficient information is available.

---

### Dependency Considerations

Identify dependencies that may affect delivery, including:

- Client approval
- Design assets
- Existing systems
- External APIs
- Third-party services

---

## 7. Pricing Guidance

### Pricing Factors

Consider factors such as:

- Project complexity
- Feature count
- Platform coverage
- Integration requirements
- Custom development effort

---

### What Affects Estimates

Estimated pricing may increase due to:

- Multiple platforms
- Advanced business logic
- Large data models
- External integrations
- AI functionality

---

### When Not to Estimate

Avoid providing estimated pricing when:

- Core project scope is unclear.
- Major features remain undefined.
- Multiple project interpretations are equally plausible.

Instead, recommend further clarification.

---

## 8. Common Assumptions

### Safe Assumptions

Unless stated otherwise, proposals may reasonably assume:

- Standard responsive design
- Modern browser compatibility
- Basic authentication where required by the requested features
- Standard deployment practices

---

### Assumptions Requiring Confirmation

Do not assume:

- Exact technology stack
- Hosting provider
- Third-party services
- Number of revisions
- Maintenance duration
- Performance targets
- Security or compliance requirements

These should be confirmed with the client.

---

## 9. Out-of-Scope Requests

Do not include services outside software development unless explicitly requested and supported.

Examples include:

- Logo or branding design
- Marketing strategy
- Advertising campaigns
- Legal consultation
- Accounting services
- Hardware procurement
- Network installation

When mixed-domain requests are identified, generate the proposal only for the software development portion and identify remaining requests separately.
