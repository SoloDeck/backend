# ADR-003: PostgreSQL as Primary Database

**Status:** Accepted

---

## Context

SoloDesk stores multi-tenant relational data: users, clients, deals, proposals, contracts, invoices, reminders. The data model has clear relational structure with foreign keys, soft-delete patterns, and JSONB fields for embedded value objects (client snapshots, proposal content).

## Problem

Which database should serve as the system of record?

## Decision

Use **PostgreSQL 16** as the sole operational database.

- All tables use `UUID` primary keys generated by `gen_random_uuid()`.
- `updated_at` fields are maintained by a PostgreSQL trigger (`set_updated_at()`), not the ORM, ensuring consistency even when rows are updated outside the application.
- JSONB is used for embedded value objects (`client_snapshot`, proposal `content`, contract `content`) where schema flexibility is needed.
- Partial unique indexes enforce business rules at the DB level (e.g., one active contract per deal).
- Soft delete via `deleted_at TIMESTAMPTZ` on mutable tables.
- Alembic manages schema migrations in async mode.
- SQLAlchemy 2.x async with `asyncpg` driver.

## Alternatives Considered

| Alternative | Reason Rejected |
|-------------|-----------------|
| MySQL / MariaDB | Weaker JSON support; partial indexes less ergonomic; no `gen_random_uuid()` |
| MongoDB | Schema flexibility not needed at this scale; joins across documents are expensive |
| CockroachDB | Distributed overhead unnecessary for current scale; higher operational complexity |
| SQLite | Not suitable for multi-process (Celery workers + API) concurrent writes |

## Consequences

**Positive:**
- Strong ACID guarantees across all business objects.
- JSONB allows schema evolution for embedded value objects without migrations.
- Partial unique indexes enforce complex constraints cheaply.
- pg trigger for `updated_at` is bulletproof — can't be bypassed by raw SQL updates.
- `asyncpg` is the fastest Python PostgreSQL driver.

**Negative:**
- Schema migrations are required for structural changes (mitigated by JSONB for volatile fields).
- Horizontal write scaling requires Citus or logical replication (not needed now).
- Soft delete requires `WHERE deleted_at IS NULL` on every query — easy to forget (enforced by code review + CLAUDE.md rules).
